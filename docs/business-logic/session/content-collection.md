# 콘텐츠 수집과 분석

> 관련 코드: `internal/services/content_collector.go`, `internal/services/stage_collect.go`

## 개요

콘텐츠 수집은 Run의 핵심 단계로, **외부 소스에서 원시 데이터를 수집**하고 **LLM이 이를 분석해 콘텐츠 앵글을 제안**하는 과정이다.

전체 흐름:
```
수집 시작
  │
  ├─ 1. 자동 리서치 소스 주입 (컨텍스트에 프롬프트가 있으면)
  ├─ 2. 각 소스별 순차 fetch
  ├─ 3. 소스 카운트 업데이트
  ├─ 4. 전체 실패 시 → error 상태로 전환, 종료
  │
  ├─ 5. analyzing 상태로 전환
  ├─ 6. LLM 분석 실행 (120초 타임아웃)
  │
  └─ 7. pending_review 상태로 전환
```

## 수집 단계

### 소스 타입과 매핑

사용자가 설정한 소스 타입은 내부 fetcher 타입으로 변환된다:

| UI 소스 타입 | 내부 fetcher | 동작 |
|-------------|------------|------|
| `rss` | rss | URL에서 RSS 피드 파싱 |
| `hn` | rss | HN RSS 피드로 변환. `min_score`가 있으면 `?points=N` 필터 |
| `reddit` | rss | Reddit RSS 피드로 변환. `subreddit` 미지정 시 `/r/all` |
| `google_trends` | rss | Google Trends RSS 피드. `geo` 미지정 시 US |
| `http` | http | URL에서 본문 텍스트 추출 |
| `social` / `twitter` | social | 키워드/계정 기반 소셜 미디어 크롤링 |
| `research` | research | LLM 기반 딥리서치. `depth`가 `light` 또는 `deep` |

### 자동 리서치 소스

Session 컨텍스트에 **프롬프트(task prompt)가 설정되어 있으면**, 사용자가 명시적으로 리서치 소스를 추가하지 않았더라도 **자동으로 리서치 소스가 맨 앞에 추가**된다:
- 리서치 깊이: 컨텍스트의 `research_depth` (기본값: `deep`)
- 리서치 모델: 컨텍스트의 `research_model`, 없으면 Session 모델

### 소스 Fetch 기록

각 소스의 수집 결과는 **SourceFetch** 레코드로 저장된다:
- 성공 시: 수집된 아이템 목록 (`RawItems`)과 건수
- 실패 시: 에러 메시지 기록 (다른 소스 수집은 계속 진행)

### 에러 처리

- 개별 소스 실패: 에러를 기록하고 **다음 소스로 계속 진행**
- 전체 소스 실패 (수집된 아이템이 0이고 에러가 있을 때): **error 상태로 전환**하고 분석을 건너뜀
- 수집할 소스가 없을 때: 바로 **pending_review로 전환**

### 취소

수집/분석 진행 중에 사용자가 Run을 취소하면(상태를 `draft`로 변경):
- 각 소스 fetch 전에 취소 여부를 확인
- 분석 시작 전에 취소 여부를 확인
- 분석 중에도 취소 여부를 확인
- 취소가 감지되면 현재 작업을 중단하고 리턴 (상태는 이미 `draft`로 변경되어 있음)

## 분석 단계

### 입력 구성

LLM에게 전달되는 프롬프트는 두 부분으로 구성된다:

**시스템 프롬프트**: `content-analyze` 스킬에서 로드 (skills 레지스트리 사용)

**유저 프롬프트**: 다음 정보를 조합:
1. **파이프라인 컨텍스트**: 프롬프트, 언어, 세션 이름/설명
2. **선호 워크플로우**: Session에 연결된 워크플로우 목록
3. **사용 가능한 워크플로우**: 시스템 내 전체 워크플로우 목록 (이름, 설명, 노드 구성)
4. **수집된 아이템**: 각 소스에서 수집된 콘텐츠 (제목, URL, 본문 500자 제한)

### 분석 결과

LLM은 JSON 형식으로 응답하며, 다음을 포함한다:

| 필드 | 설명 |
|------|------|
| `summary` | 수집된 콘텐츠의 종합 요약 |
| `insights` | 핵심 인사이트 목록 |
| `suggested_angles` | 콘텐츠 제작 앵글 제안 (포맷, 헤드라인, 추천 워크플로우, 근거) |
| `overall_score` | 전체 콘텐츠 품질 점수 |

### 워크플로우 매칭

분석에서 제안된 각 앵글의 워크플로우는 실제 존재 여부가 검증된다:
- **존재하는 워크플로우** → `match_type: "matched"`
- **존재하지 않는 워크플로우** → 워크플로우 이름을 비우고 `match_type: "none"`
- 사용자가 수동으로 매핑할 수 있다 → `match_type: "manual"`
- 워크플로우 자동 생성을 요청할 수 있다 → `match_type: "generated"`

### 분석 재시도

`analyzing` 상태에서 멈춘 세션에 대해 분석만 다시 실행할 수 있다. 수집 단계를 건너뛰고 기존 fetch 데이터를 사용해 LLM 분석을 재실행한다.

## 분석 결과 편집

사용자는 리뷰 단계에서 분석 결과를 수정할 수 있다:
- **요약/인사이트 수정**: summary와 insights를 직접 편집
- **앵글-워크플로우 매핑**: 특정 앵글에 워크플로우를 수동 지정
- **앵글용 워크플로우 생성**: 앵글 설명을 기반으로 LLM이 새 워크플로우를 자동 생성하고 저장
