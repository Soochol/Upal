# 프로젝트 플래닝 Skill 설계

## 목적

코드 1줄 쓰기 전에 프로젝트의 80%를 문서로 플래닝하는 Claude Code skill 시스템.
새 프로젝트 시작과 기존 프로젝트 대규모 기능 추가 둘 다 지원한다.

## 핵심 원칙

- **5개 질문**: 무슨 문제? → 뭐가 있나? → 어떻게 움직이나? → 코드를 어떻게 정리? → 뭐부터?
- **Phase별 독립 skill**: 세션을 나눠 진행 가능, 컨텍스트 윈도우 문제 없음
- **누적 읽기**: 각 Phase는 이전 Phase의 결과 문서(압축본)를 읽어서 맥락을 이어감
- **결과물은 한국어**

## 구조

### Skill 파일 (6개)

```
.claude/commands/
├── plan-project.md       # 오케스트레이터
├── plan-problem.md       # Phase 1: 무슨 문제를 푸는가
├── plan-domain.md        # Phase 2: 뭐가 있나
├── plan-flow.md          # Phase 3: 어떻게 움직이나
├── plan-architecture.md  # Phase 4: 코드를 어떻게 정리하나
└── plan-roadmap.md       # Phase 5: 뭐부터 만드나
```

### 결과 문서 (Phase당 1개)

```
docs/plans/{project-name}/
├── 01-problem.md
├── 02-domain.md
├── 03-flow.md
├── 04-architecture.md
└── 05-roadmap.md
```

## 각 Skill 상세

### `/plan-project` — 오케스트레이터

**역할**: 진행 상황 추적 + 다음 단계 안내

**동작**:
1. `docs/plans/` 하위 폴더를 스캔
2. 각 Phase 결과 파일 존재 여부로 진행 상태 판단
3. 현황을 보여주고 다음 Phase를 안내
4. 기존 프로젝트면 코드베이스를 분석해서 건너뛸 수 있는 Phase 제안

**출력 예시**:
```
프로젝트: upal

  ✅ Phase 1: 문제 정의 (01-problem.md)
  ✅ Phase 2: 도메인 모델 (02-domain.md)
  ⬜ Phase 3: 비즈니스 흐름
  ⬜ Phase 4: 아키텍처
  ⬜ Phase 5: 구현 계획

→ 다음: /plan-flow 를 실행하세요.
```

### `/plan-problem` — Phase 1: 무슨 문제를 푸는가

**읽는 파일**: 없음 (첫 Phase)

**질문 흐름** (한 번에 하나씩):
1. 이 소프트웨어를 쓰는 사람은 누구인가?
2. 그 사람이 지금 겪는 문제/불편은 뭔가?
3. 이 소프트웨어로 뭘 할 수 있게 되는가? (핵심 시나리오 3-5개)
4. 이번에 만들 것과 안 만들 것의 경계는?
5. 성공했다는 걸 어떻게 아는가?

**기존 프로젝트 모드**: 코드베이스와 README를 분석해서 초안을 먼저 제시하고, 사용자가 수정/보완.

**출력 (`01-problem.md`)**:
```markdown
# 문제 정의

## 대상 사용자
(누가 쓰는가)

## 핵심 문제
(왜 필요한가)

## 사용자 시나리오
- 시나리오 1: (사용자가 X를 한다 → Y가 된다)
- 시나리오 2: ...

## 범위
### 포함
- ...
### 제외
- ...

## 성공 기준
- ...
```

### `/plan-domain` — Phase 2: 뭐가 있나

**읽는 파일**: `01-problem.md`

**질문 흐름**:
1. 시나리오에서 반복적으로 등장하는 핵심 개념(명사)은?
2. 각 개념이 독립적인가, 다른 것에 종속되는가?
3. 개념 간 관계는? (X는 Y를 여러 개 가진다, X는 Y에 속한다)
4. 자연스럽게 묶이는 그룹이 있는가? (경계)
5. 팀에서 이 개념을 뭐라고 부르는가? (용어 통일)

**기존 프로젝트 모드**: 코드의 주요 타입/모델을 분석해서 도메인 모델 초안을 먼저 제시.

**출력 (`02-domain.md`)**:
```markdown
# 도메인 모델

## 핵심 개념

| 개념 | 설명 | 독립/종속 |
|------|------|----------|
| ... | ... | ... |

## 관계도

(ASCII 다이어그램)

## 경계 (그룹)

### 그룹 1: {이름}
- 포함 개념: ...
- 책임: ...

### 그룹 2: ...

## 용어 사전

| 용어 | 의미 | 비고 |
|------|------|------|
| ... | ... | ... |
```

### `/plan-flow` — Phase 3: 어떻게 움직이나

**읽는 파일**: `01-problem.md` + `02-domain.md`

**질문 흐름**:
1. 핵심 시나리오별로: 사용자가 X를 하면 시스템에서 무슨 일이 일어나는가?
2. 각 단계에서 어떤 개념의 상태가 바뀌는가?
3. 중간에 실패하면 어떻게 되는가?
4. 자동으로 일어나는 것 vs 사용자가 직접 하는 것?

**출력 (`03-flow.md`)**:
```markdown
# 비즈니스 흐름

## 흐름 1: {시나리오 이름}

### 정상 흐름
1. 사용자가 ...
2. 시스템이 ...
3. ...

### 상태 전환
(ASCII 상태 다이어그램)

### 예외/실패
- ...하면 → ...로 처리

## 흐름 2: ...
```

### `/plan-architecture` — Phase 4: 코드를 어떻게 정리하나

**읽는 파일**: `01-problem.md` + `02-domain.md` + `03-flow.md`

**질문 흐름**:
1. 기술 스택 제약이 있는가? (언어, 프레임워크, DB)
2. (없으면) 도메인 특성에 맞는 스택을 제안하고 선택
3. 코드를 어떤 레이어/폴더로 나눌 건가?
4. 레이어 간 규칙은? (뭐가 뭘 호출할 수 있는가)
5. 외부 인터페이스는? (API 엔드포인트, 이벤트)

**기존 프로젝트 모드**: 현재 폴더 구조와 아키텍처를 분석해서 "현재 구조" + "추가/변경 제안"으로 제시.

**출력 (`04-architecture.md`)**:
```markdown
# 아키텍처

## 기술 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| 언어 | ... | ... |
| 프레임워크 | ... | ... |
| DB | ... | ... |

## 폴더 구조
(트리 다이어그램)

## 레이어별 책임과 규칙

### {레이어 이름}
- 책임: ...
- 규칙: ...
- 의존 방향: ...

## API 설계

| 메서드 | 경로 | 설명 |
|--------|------|------|
| ... | ... | ... |

## 데이터 모델
(주요 테이블/스키마)

## 기술 결정 기록
### 결정 1: {제목}
- 선택: ...
- 대안: ...
- 이유: ...
```

### `/plan-roadmap` — Phase 5: 뭐부터 만드나

**읽는 파일**: 모든 이전 Phase 문서 (01~04)

**질문 흐름**:
1. 가장 먼저 동작해야 하는 핵심 기능은?
2. 각 기능의 의존 관계는?
3. 각 마일스톤에서 "동작하는 소프트웨어"가 나오는가?
4. 불확실하거나 리스크가 있는 부분은?

**출력 (`05-roadmap.md`)**:
```markdown
# 구현 계획

## 마일스톤

### M1: {이름}
- 목표: ...
- 작업 목록:
  - [ ] ...
  - [ ] ...
- 완료 기준: ...
- 선행 조건: 없음

### M2: {이름}
- 목표: ...
- 선행 조건: M1

## 의존관계

(ASCII 다이어그램)

## 리스크 / 불확실한 것
- ...
```

```
/plan-problem  ──저장──▶ 01-problem.md
                              │
/plan-domain   ──읽기──◀──────┘──저장──▶ 02-domain.md
                                             │
/plan-flow     ──읽기──◀─────────────────────┘──저장──▶ 03-flow.md
                                                            │
/plan-architecture ──읽기──◀────────────────────────────────┘──저장──▶ 04-architecture.md
                                                                           │
/plan-roadmap      ──읽기──◀───────────────────────────────────────────────┘──저장──▶ 05-roadmap.md
```

## 공통 규칙

1. **질문은 한 번에 하나씩**. 객관식 선호.
2. **기존 프로젝트 모드**: 코드를 먼저 분석해서 초안을 제시하고, 사용자가 수정.
3. **결과물은 한국어**.
4. **저장 전 반드시 사용자 확인**. "이 내용으로 저장할까요?"
5. **Phase 건너뛰기 가능**. 오케스트레이터에서 안내.
6. **재실행 가능**. 같은 Phase를 다시 실행하면 기존 문서를 덮어쓸지 물어봄.
