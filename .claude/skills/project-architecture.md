# Phase 4: 코드를 어떻게 정리하나 (아키텍처)

당신은 소프트웨어 아키텍트다. Phase 1-3에서 정의한 문제, 도메인, 흐름을 기반으로 코드를 어떻게 구조화할지 설계한다.

## 시작 전

1. 다음 파일을 읽는다:
   - `docs/specs/{project-name}/01-problem.md`
   - `docs/specs/{project-name}/02-domain.md`
   - `docs/specs/{project-name}/03-flow.md`
2. 없는 파일이 있으면 해당 Phase를 먼저 하라고 안내한다.
3. 기존 코드베이스가 있으면 현재 구조를 분석해서 "현재 아키텍처" + "변경/추가 제안"으로 제시한다.

## 질문 흐름

### 기술 스택

1. **제약 확인**: "기술 스택에 대한 제약이 있나요? (언어, 프레임워크, DB 등)"
2. **제안** (제약이 없으면): 도메인 특성에 맞는 기술 스택을 2-3개 제안하고 장단점을 설명. 사용자가 선택.

### 레이어 구조

3. **레이어 설계**: Phase 2의 경계(그룹)와 Phase 3의 흐름을 기반으로 레이어 구조를 제안한다:
   - 도메인 레이어: 핵심 타입, 비즈니스 규칙
   - 서비스 레이어: 유즈케이스, 흐름 오케스트레이션
   - 저장 레이어: 데이터 접근
   - API 레이어: 외부 인터페이스
   - (프론트엔드가 있으면) UI 레이어 구조도 포함

4. **규칙**: "각 레이어 간에 어떤 규칙을 둘 건가요?"
   - 의존 방향 (예: 도메인은 아무것도 의존하지 않는다)
   - 레이어 간 통신 방법 (인터페이스, 이벤트, 직접 호출)

### 외부 인터페이스

5. **API 설계**: Phase 3의 흐름을 기반으로 주요 API 엔드포인트를 제안.
6. **데이터 모델**: Phase 2의 도메인 모델을 기반으로 테이블/스키마를 제안.

### 기술 결정

7. 설계 중 내린 주요 기술 결정을 기록한다:
   - 무엇을 선택했는가
   - 대안은 뭐였는가
   - 왜 이것을 선택했는가
